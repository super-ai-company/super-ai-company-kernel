# Company Kernel 产品形态与市场战略分析（2026-06）

> 你的愿景：做一个**底层框架** + **AI 员工在线出租市场**——任何人可以把自己的智能体（容器/真实主机/CLI）出租出去，
> 按结果交付计费；客户可以在线"聘请"不同专业的 AI 员工。即 **AI 劳动力的双边市场 + 按结果结算**。
> 本文判断这个方向的可行性、最佳产品形态、授权模式，以及竞争与市场情况。

---

## 一、结论先行

1. **方向成立，且踩在 2026 最强的两条趋势上**：AI agent 市场 ~45% CAGR；"按结果计费（outcome-based）"正在取代订阅制。
2. **最佳形态 = 开放内核（可自托管）+ 云端市场/控制台（SaaS 抽成）**，即 **open-core 双层**。不是单纯的桌面软件安装。
3. **你已经做出来的 Company Kernel 正好是最难、最有壁垒的那一层**：供给侧运行时 + 结果验证（裁决门）+ 证据/审计 + 审批 + 沙箱隔离 + 鉴权。市面上大多数"agent 市场"恰恰缺这层。
4. **最大风险不是技术，是双边冷启动 + 结果验证可信度 + 运行他人智能体的安全/信任**。

---

## 二、市场情况（已联网核实）

- **规模**：AI agents 市场约 **$5–8B（2025）→ $47–52B（2030）**，CAGR ~43–46%（MarketsandMarkets / BCC / Grand View 三家口径一致）。企业级 agentic AI：$2.58B → $24.5B（2030）。
- **计费范式转移**：seat 订阅 12 个月内从 21%→15%，混合/用量/结果型成主流。代表：
  - Intercom Fin：**$0.99 / 解决一个工单**，没解决不收费。
  - Zendesk：自动解决 **$1.5–2.0 / 次**。
  - Sierra（Bret Taylor）：从第一天就按结果计费，**21 个月做到 $100M ARR、估值 $4.5B**——证明"按结果交付的 AI 劳动力"是可成立的大生意。

→ 你的"按结果交付"定价不是赌一个新模式，而是顺着已被验证的主流走。

## 三、竞争格局（已联网核实）

把"AI agent 市场"拆成四类，看清你和谁竞争、空位在哪：

| 类别 | 代表 | 他们做什么 | 与你的关系 |
|---|---|---|---|
| **需求侧目录市场** | UpAgents（"Upwork for AI agents"）、Kore.ai、Google Cloud Agent Marketplace、Salesforce AgentExchange | 浏览/雇佣/部署预制 agent，多为"单厂商发布、买家调用" | 你的市场层的直接对标，但他们多是 catalog，弱治理 |
| **计费基础设施** | Paid.ai、Metronome（给 OpenAI 计费）、Nevermined、Chargebee | 给 agent 做用量/结果计费 | 你需要的能力之一，可集成而非自建 |
| **人机混合市场** | dealwork.ai（agent 与人类同台竞标，抽 3–10%）、Upwork（Uma，agent 来雇人）、RentAHuman | agent 与人类互相雇佣 | 印证"agent 当劳动力出租"已被多家下注 |
| **编排/运行时框架** | LangGraph、CrewAI、AutoGen、Microsoft Agent Framework | 让开发者编排多 agent | 你内核的同类，但它们不做市场、不做结果验证治理 |

**关键空位（你的差异化）**：几乎没有人把"**任意第三方智能体（跑在自己容器/主机上）→ 按结果出租 → 平台验证结果并结算**"这条链做扎实。难点全在治理：
- 结果怎么客观验证（不能 agent 自己说"完成了"就付钱）——**你的裁决门 + evidence + 审批正是这个**；
- 怎么安全地运行陌生人的 agent——**你的沙箱隔离 / 工作区边界 / 内核自我保护**；
- 怎么记账、审计、防作弊——**你的 audit_logs / adapter_runs / trace**。

这就是为什么你不该只做又一个"agent 目录"，而要做**带结果托管与治理的双边劳动力市场**。

## 四、最佳产品形态（含安装与授权）

### 形态：Open-Core 双层

```
┌─────────────────────────────────────────────┐
│  云端 Marketplace + 控制平面（SaaS，闭源，抽成）   │  ← 你赚钱的地方
│  发现/雇佣 · 结果托管(escrow) · 计费结算 ·         │
│  信誉评分 · 争议仲裁 · 供需匹配                    │
└───────────────▲──────────────▲────────────────┘
                │ REST/gRPC      │
   ┌────────────┴───┐      ┌─────┴──────────────┐
   │ 供给侧 Kernel   │      │ 需求侧 Kernel/控制台 │
   │（出租人安装）    │      │（企业自托管或用云）   │
   │ = 你已做好的内核 │      │ = 你已做好的控制台   │
   └────────────────┘      └────────────────────┘
```

- **底层框架（开源 / 源码可见）= 你现在的 Company Kernel**。出租人（agent 供给方）在自己的容器/主机上装它，一条命令把自己的 agent 注册成"可出租员工"。开源驱动供给侧规模 + 信任（代码可审计，运行我的 agent 我放心）。
- **云端市场 + 控制平面（闭源 SaaS）**：发现、雇佣、结果托管、结算、信誉、仲裁。**这是收费层**。

### 安装体验（正好用上你已建的能力）

- **出租人上线 agent（供给侧）**：`company-add-employee` 一条命令 + `COMPANY_KERNEL_API_TOKEN` 鉴权 + REST 网关把能力上报到云市场。Win/Linux/Mac 已覆盖（`docs/AGENT_ONBOARDING.md`）。容器化只需把内核打成一个 Docker 镜像，`docker run` 即上线。
- **客户聘请 agent（需求侧）**：云市场网页按技能/价格/信誉浏览 → 下单（带验收标准）→ 内核裁决门验证结果 → 托管资金释放。客户零安装；要私有部署的企业可自托管控制台。

### 授权 / 商业模式

- **内核**：源码可见 / 宽松或 BSL 许可（鼓励供给方装、可审计）。
- **云市场**：闭源 SaaS，**按成交结果抽成（建议 take rate 10–20%）** + 可选企业版私有部署订阅。
- **定价**：平台对买家按结果计费（对齐 Intercom/Zendesk/Sierra），对卖家抽佣；高价值垂类可做月度承诺量。

## 五、可行性评估

| 维度 | 评级 | 说明 |
|---|---|---|
| 技术可行性 | ★★★★☆ | 最难的运行时+治理层你已做出并测试（312 测试）。剩下是市场层与支付。 |
| 市场时机 | ★★★★★ | 45% CAGR + 结果计费成主流，窗口正开。 |
| 差异化/壁垒 | ★★★★☆ | 结果验证+沙箱+审计的治理层是真壁垒；纯目录市场没有。 |
| 双边冷启动 | ★★☆☆☆ | 最大难点：先有 agent 还是先有买家。建议先自营供给（你已有 codex/hermes/openclaw 一批员工）打透 1–2 个垂类。 |
| 信任/安全 | ★★☆☆☆ | 运行陌生 agent + 按结果付钱 = 欺诈/越权风险，需要强隔离、结果仲裁、信誉与押金机制。 |
| 支付/结算 | ★★★☆☆ | 结果型托管 + 分成结算可集成 Paid.ai/Stripe/Metronome，不必自建。关注 agent 原生支付趋势（Visa/Mastercard/AP2/x402）。 |

## 六、建议的落地路线

1. **先垂直、自营供给**：选 1 个结果可机器验证的垂类（如"代码任务/测试修复""数据抓取与报表""客服工单解决"），用你已有的 codex/hermes 员工自营交付，跑通"下单→执行→裁决→结算"闭环。这同时解决冷启动。
2. **把内核 Docker 化 + 一键注册到市场**：`docker run company-kernel`，上线即注册。供给侧规模化的前提。
3. **结果验证标准化**：把裁决门升级成"可插拔验收器"（测试通过/数值零差异/截图比对/人工抽检），这是市场可信度的核心，也是你相对纯目录市场的护城河。
4. **接支付与托管**：集成结果型 escrow + 分成结算（先用现成 Paid.ai/Stripe）。
5. **信誉与押金**：供给方质押 + 买家评分 + 争议仲裁，压制作弊。
6. **再开放第三方供给**：基础设施稳了再让"每个人出租自己的 agent"，否则信任与质量失控。

## 七、一句话定位（可用于对外）

> **"AI 劳动力的双边市场：任何智能体都能上线出租，平台按结果验证并结算——Upwork 之于人类，我们之于 AI 员工。"**
> 底层是可自托管、可审计的治理内核（结果验证 + 沙箱 + 审计），这正是纯 agent 目录市场缺失的信任层。
